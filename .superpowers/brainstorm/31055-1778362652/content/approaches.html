Three approaches to building the app

All use base R for the simulation engine. The difference is code structure and UI polish.

📁 poker_simulator/
└── app.R
# ui, server, and poker logic
# all in one file (~300 lines)

A — Single file app.R

Everything in one place. Easy to read top-to-bottom. Good for learning but gets hard to navigate as the app grows.

Pros

  • Simplest to follow
  • One file to deploy

Cons

  • Long file, harder to edit
  • Logic and UI are mixed
📁 poker_simulator/
├── app.R # entry point
├── ui.R # layout & inputs
├── server.R # reactive logic
└── simulation.R # poker math
simulation.R handles:
deal_cards(), eval_hand(),
run_simulation()

B — Split files (Recommended)

UI, server logic, and poker math each live in their own file. Each file is focused and short. This is the standard pattern for Shiny apps on Posit Connect.

Pros

  • Clean separation of concerns
  • Poker logic reusable & testable
  • Standard Posit Connect pattern

Cons

  • More files to navigate

C — Split files + bslib UI

Same structure as B, but uses the bslib package for a modern Bootstrap 5 look — polished value boxes, card layouts, and theming out of the box.

Pros

  • Most professional-looking
  • Great on Posit Connect

Cons

  • Extra package dependency
  • More to learn upfront
