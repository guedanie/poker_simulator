Three approaches for the Python Shiny app

All use Core mode. Difference is file structure and UI polish.

📁 poker_simulator/
└── app.py # ~300 lines, everything here
# app.py contains:
def deal_cards(): ...
def eval_hand(): ...
def run_simulation(): ...
app_ui = ui.page_fluid(...)
def server(input, output, session): ...
app = App(app_ui, server)

A — Single app.py

UI, server logic, and poker simulation all in one file. Fine for a first pass, but gets unwieldy fast.

Pros

  • Simple — one file to open
  • Easy to deploy

Cons

  • Long and hard to navigate
  • Poker logic buried in app code
📁 poker_simulator/
├── app.py # App(app_ui, server)
├── ui.py # all ui.* layout code
├── server.py # reactive logic
└── simulation.py # poker math
# simulation.py — pure Python, no Shiny
def parse_card(s): ...
def eval_hand(cards): ...
def run_simulation(hole, n_players, n_sims): ...

B — Split files Recommended

Poker math lives in simulation.py — completely independent of Shiny. UI and server are in their own files. Each file is short and focused.

Pros

  • Clean, navigable structure
  • Poker logic easy to test alone
  • Standard Posit Connect pattern

Cons

  • More files to open initially
📁 poker_simulator/
├── app.py
├── ui.py # uses shinyswatch theme
├── server.py
└── simulation.py
# ui.py — polished look
import shinyswatch
ui.page_fluid(
shinyswatch.theme.darkly(),
# dark casino-style theme
)

C — Split files + shinyswatch

Same structure as B, but adds shinyswatch for a polished dark theme (think casino aesthetic). One extra dependency, looks great on Posit Connect.

Pros

  • Professional look out of the box
  • Dark theme fits poker well

Cons

  • Extra package to install
  • Slightly more complex UI code
